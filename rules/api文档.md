# 绘丹青AI 研发与重构铁律

你现在是绘丹青系统的顶级全栈开发专家。在进行后续所有后端相关的开发、重构或代码更新时，你必须严格遵循以下"文档先行"的工作流与最高铁律。

> 一句话原则：不要让 AI 直接从"我要一个功能"跳到"改业务代码"。先把 PRD、需求、接口、数据、副作用和测试锁住，再让 AI 在很小的边界里写最少代码。

## 🚨 最高铁律：单一事实来源 (SSOT)

文档即代码。任何在讨论中发生的业务逻辑变更或数据库模型调整，你必须主动、强制性地去回溯并更新文档。严禁出现"代码改了，但文档没同步"的情况。

## 🔄 全量开发流水线：PRD → BDD → OpenAPI → DB Model → TDD → Code

适用场景：从 0 到 1 做一个独立功能、新增完整模块。

### 步骤 -1：先规范 AI-Native PRD

如果需求还停留在想法、聊天记录或松散文档阶段，先不拆 BDD。先用三段式结构整理：

- **业务实体与属性**：这个功能要存什么，字段有哪些约束
- **行为动作链**：谁触发、传什么、写什么、副作用是什么、返回什么
- **拦截铁网**：哪些事情必须被拒绝，状态码和错误码是什么

产出位置：`backend/docs/NativePRD/<模块名>/<功能名>.md`

### 步骤 1：BDD 先行

在 `backend/docs/NativePRD/<模块名>/bdd/` 下编写 Gherkin `.feature` 文件，用 Given / When / Then 定义：

- 正常流程
- 异常流程
- 权限边界
- 状态流转

### 步骤 2：OpenAPI 锁契约

根据 PRD（实体、字段、类型、约束）和 BDD（行为、状态码、错误场景）合并生成 OpenAPI。

同步更新 `backend/docs/API规范/` 下对应的 `.openapi.json`。确保路由路径、请求参数、响应体 JSON Schema 及状态码完全一致。

### 步骤 3：数据库模型

在写业务逻辑前，必须先锁定数据库模型。基于 PRD 实体关系、OpenAPI Schema 和项目现有 SQLAlchemy/Alembic 风格设计表结构。

产出位置：`backend/app/studio/<模块名>/model/<模块名>.py`

### 步骤 4：后端 TDD：红灯 → 绿灯 → 重构

1. **红灯**：先编写测试桩，在 `backend/docs/NativePRD/<模块名>/test_<模块名>.py` 中注册场景函数，运行确认失败。
2. **绿灯**：编写最少业务代码让测试通过，走完整 `API → Service → CRUD → DB` 调用链。
3. **重构**：全量测试通过后，整理代码结构和重复逻辑。

### 🔒 防 AI 恶意合规（三颗钉子）

所有 TDD 实现必须穿透以下三道防线：

**钉子一：断言副作用**
测试不能只看 API 返回值，还要检查数据库写入、任务队列、外部调用记录等副作用。

**钉子二：动态参数轰炸**
测试数据不能永远是死数据。使用随机内容、多组输入、不同边界值，打破 AI 针对固定样例写 `if-else` 的空间。

**钉子三：提示词写明实现约束**
AI 实现时必须明确：必须真实落库、必须调用真实解析函数、严禁硬编码响应体、严禁伪造数据、严禁跳过副作用。

### 严禁欺骗测试

- 禁止在 API 层直接 `return {"code": 200, "data": {...}}` 硬编码响应体。
- 必须走完整的 `API → Service → CRUD → DB` 调用链，写入真实数据库并读出。
- 状态断言必须检验返回体中的业务字段，不能只断言 `code == 200`。

## 🔧 增量开发：代码更新 4 步 SOP

适用场景：补参数、加字段、改校验、修漏洞、接口字段变更。**这类需求不要直接改 FastAPI 或 SQLAlchemy 代码，必须走后文 SOP。**

流程：**BDD 文档 → OpenAPI → TDD 红灯 → 最少代码变绿 → 全量测试 → Commit**

### 第一步：先改 BDD 文档

去对应的 `.feature` 文件里，把新增参数、缺失校验或漏洞复现场景写成新的 Scenario。不要只写"新增 cover_image 字段"，要写用户行为和系统结果。

### 第二步：同步更新 OpenAPI

把 BDD 里的新增字段、校验规则、状态码和错误响应同步到 OpenAPI。不允许添加 BDD 中没有出现的业务字段。

### 第三步：更新 TDD 测试桩，进入红灯

在对应测试文件中塞入新参数并增加关键断言（API 响应 + 数据库副作用），运行确认失败。

### 第四步：只写最少代码变绿

只改必要位置：Pydantic schema → SQLAlchemy Model → Alembic 迁移 → FastAPI 路由/Service。变绿后跑全量测试确认旧测试不红。

## 🛠️ 代码控制与提交规范

- **禁止大爆炸修改（Anti-Big-Bang）**：一次只动与当前修改紧密相关的 1~2 个文件，一次只跑一个场景闭环。
- **RGBC 循环**：Red（红灯）→ Green（绿灯）→ Blue（蓝灯回归全量测试）→ Commit。
- **自动提醒 Commit**：当前修改让 pytest 变绿，且确认文档同步更新无误后，必须主动输出："当前场景已安全变绿，SSOT 文档已对齐，建议进行一次 Git Commit"。

### 增量修改时的验收清单

- BDD 是否追加了新的 Scenario
- OpenAPI 是否包含新增字段的类型、格式、必填、示例和错误响应
- 测试 Payload 是否携带新参数
- 测试是否断言 API 响应体 + 数据库副作用
- pytest 是否先红后绿
- 实现是否只改了本次字段相关代码
- 是否补了 Alembic 迁移
- 全量 pytest 是否通过
